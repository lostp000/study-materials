# HW14 – эмбеддинги, FAISS, оценка retrieval и mini-RAG по базе знаний

## 1. Кратко: что сделано

- **База знаний**: 10 документов по теме "Основы машинного обучения" (линейная регрессия, SVM, Random Forest, градиентный бустинг, метрики, регуляризация и др.). Выбрана потому, что тема хорошо структурирована, документы имеют чёткую тематическую принадлежность, что упрощает оценку retrieval.
- **Чанкинг**: документы разбиты на перекрывающиеся фрагменты по 100 слов с overlap 20 слов для сохранения контекста на границах.
- **Векторизация**: использована модель `all-MiniLM-L6-v2` из `sentence-transformers` с нормализацией эмбеддингов.
- **Retrieval**: базовая проверка показала hit@3 = 1.0, но hit@1 = 0.8 — не все запросы находят релевантный документ на первом месте.
- **Обновление**: добавлено 3 новых документа (K-Means, PCA, нейронные сети), после переиндексации новые запросы стали находиться корректно.
- **Mini-RAG**: реализован extractive-подход с возвратом источников. Работает хорошо для прямых вопросов, но испытывает трудности с вопросами, требующими обобщения из нескольких источников.

## 4. Retrieval и индекс

### 4.1. Эмбеддинги и индекс

- **Какая embedding-модель или способ векторизации использовались**: `all-MiniLM-L6-v2` (sentence-transformers)
- **Какая метрика сходства использовалась**: косинусное сходство (через скалярное произведение нормализованных векторов)
- **Какой индекс `FAISS` использовался**: `IndexFlatIP` (Inner Product)
- **Какой `top_k` использовался по умолчанию**: 3
- **Что показали первые примеры retrieval**: модель хорошо находит релевантные документы по ключевым терминам. Например, запрос "градиентный бустинг" вернул `doc_06` на первом месте. Однако некоторые запросы с синонимами (например, "SVM" вместо "метод опорных векторов") работают чуть хуже.

### 4.2. Контрольные запросы и оценка retrieval

- **Сколько контрольных запросов использовалось**: 10 (по одному на каждый базовый документ)
- **Что считалось релевантным ответом**: документ, соответствующий теме запроса (связь query ↔ doc_id)
- **Какие метрики retrieval считались**: hit@k (k=1,3,5), recall@k (k=1,3,5), MRR@3
- **Итоговый `hit@k`**: hit@1 = 0.8, hit@3 = 1.0, hit@5 = 1.0
- **Итоговый `recall@k`**: recall@1 = 0.45, recall@3 = 0.72, recall@5 = 0.92
- **Дополнительные метрики (если были)**: MRR@3 = 0.9
- **Какие запросы оказались самыми простыми**: "Что такое линейная регрессия?", "Как работает градиентный бустинг?" — прямые формулировки, совпадающие с терминологией документов.
- **Какие запросы оказались самыми проблемными**: "В чем разница между L1 и L2 регуляризацией?" — запрос требует сравнения, а не просто извлечения факта. "Зачем нужно масштабировать признаки?" — ответ распределён по нескольким предложениям в одном документе.

## 5. Эксперимент по параметрам retrieval и обновление базы знаний

- **Какой параметр сравнивался**: `chunk_size`
- **Какие два варианта сравнивались**: 50 слов и 150 слов (с фиксированным overlap 20)
- **Как изменилось качество retrieval**: 
  - chunk_size=50: hit@3 = 0.9 (больше чанков, но выше шум)
  - chunk_size=150: hit@3 = 1.0 (лучше сохраняется контекст, но меньше чанков)
- **Какой вариант вы выбрали как основной и почему**: chunk_size=100 как компромиссный вариант, обеспечивающий баланс между сохранением контекста и количеством чанков
- **Что именно было обновлено в базе знаний**: добавлены 3 новых документа (K-Means, PCA, нейронные сети)
- **Как была выполнена переиндексация**: полное перестроение чанков, эмбеддингов и FAISS индекса
- **Что изменилось в retrieval после обновления базы знаний**: новые запросы (про K-Means, PCA, нейронные сети) стали успешно находить соответствующие документы. Старые запросы не пострадали — их hit@3 остался 1.0.

## 7. Анализ

Выбранная база знаний оказалась удачной для учебных целей: тема ML хорошо структурирована, документы имеют чёткую тематическую принадлежность. Это позволило легко составить контрольные запросы и оценить retrieval объективно.

Чанкинг с параметрами 100/20 показал себя оптимально: достаточно большие чанки сохраняют контекст, а overlap предотвращает потерю информации на границах. Эксперимент с другими размерами чанков подтвердил, что слишком маленькие чанки (50 слов) дробят смысл, а слишком большие (150) дают меньше гибкости.

Формальная оценка retrieval оказалась очень полезной. Без метрик hit@k и recall@k можно было бы ошибочно посчитать систему идеальной, увидев несколько удачных примеров. На самом деле hit@1=0.8 показывает, что 20% запросов не находят релевантный документ на первом месте.

Эксперимент с параметрами показал, что увеличение chunk_size до 150 повышает hit@3 до 1.0, но уменьшает количество чанков, что может быть проблемой для больших документов. Обновление базы знаний продемонстрировало важность переиндексации: без неё новые документы просто не находятся.

Наиболее показательные ошибки mini-RAG возникают на вопросах, требующих сравнения или обобщения. Например, "разница между L1 и L2" — информация есть в одном документе, но в разных предложениях, и extractive-подход не может их объединить.

Ограничения текущего решения: extractive-генерация вместо генеративной LLM, отсутствие reranking, примитивная оценка качества ответа (по ключевым словам), маленький размер корпуса. В production-системе потребовались бы более мощные инструменты.

## 8. Итоговый вывод

Базовым retrieval-конвейером для данной задачи я бы оставил конфигурацию с chunk_size=100, overlap=20, эмбеддингами all-MiniLM-L6-v2 и FAISS IndexFlatIP. Это даёт hit@3=1.0 при разумных вычислительных затратах.

Главный вывод про качество retrieval: метрики (hit@k, recall@k, MRR) незаменимы для объективной оценки. Без них легко переоценить или недооценить систему.

Главный вывод про связь retrieval и mini-RAG: качество retrieval напрямую определяет качество ответа. Даже идеальный генератор не спасёт ситуацию, если retrieval вернул нерелевантный контекст. RAG — это система, где каждый компонент важен.
